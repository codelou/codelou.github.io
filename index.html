浅谈函数节流

发表于 2018-03-08

浅谈函数节流

前段时间因为项目的需要，需要写个鼠标滚动页面横向滚动的效果，这里就用到了鼠标滚动事件，又因此衍生出了函数节流。。

鼠标的mousemove、scroll，浏览器窗口的resize事件等，都是在短时间内重复触发。以onresize事件为例，若事件处理程序需要进行修改元素宽度高度等操作，那么频繁的触发事件会导致频繁的重绘页面。

DOM操作比非DOM交互需要更多的内存和CPU时间，连续尝试进行过多的DOM相关操作可能会导致浏览器挂起，有时候甚至会崩溃。为了解决这个问题，需要使用定时器对该函数进行节流。

函数节流背后的基本思想是：某些代码不可以在没有间断的情况下连续重复执行。第一次调用函数，创建一个定时器，在指定的时间间隔之后运行代码，当第二次调用函数时，它会清除前一次的定时器并设置另一个。如果前一次的定时器已经执行过了，这个操作就没有任何意义。然而，如果前一个定时器尚未执行，其实就是将其替换成一个新的定时器。目的是只有在执行函数的请求停止了一段时间之后才执行。

throttle()函数：自动地进行定时器的设置和清除。接收两个参数：要执行的函数和在哪个作用域执行。throttle()首先清除之前设置的任何定时器，定时器ID是存储在函数的tID属性中的，第一次把方法传递给throttle()函数的时候，这个属性可能并不存在。接下来，创建一个新的定时器，并将其ID存储在方法的tID属性中。如果这是第一次对这个方法调用throttle()的话，那么这段代码就会创建该属性。定时器代码使用call()来确保方法在适当的环境中执行。如果没有给出第二个参数，那么就在全局作用域内执行该方法。

function throttle(method,context){
 clearTimeout(method,tId);
 method.tId = setTimeout(function(){
  method.call(context); 
 },500);
}

时间间隔设为500ms，这表示监听鼠标的mousemove、scroll，浏览器窗口的resize等事件，事件处理函数会连续调用多次，但只在最后一次触发该事件。

n=0;
window.onresize = function(){
 console.log(n);
 n++; 
};

问题：简单的一次改变窗口大小就会导致函数不停地被调用，直到窗口大小不再变化。
解决：利用定时器，让函数执行延迟500毫秒，在500毫秒内如果有函数又被调用则删除上一次调用，这次调用500毫秒后执行，如此往复。

n=0;
function resizeHandler(){
 console.log(n);
 n++;
}    
function throttle(method,context){
 clearTimeout(method,tId);
 method.tId = setTimeout(function(){
  method.call(context);
 },500);
}
window.onresize = function(){
 throttle(resizeHandler,window);
}

mvvm 设计模式

发表于 2017-07-07

MVVM 设计模式

前言

相信学习编程的同学一定听过MVC，MVP，MVVM等这样的一些词，其实它就是一种编程的设计模式，这些模式也是随着前端(这里只讨论前端)技术不断发展衍生出来的，这里我们这种介绍mvvm。

MVVM 是Model-View-ViewModel的缩写,是流行于前端开发的一种架构模式，它的核心是提供了view和viewModel的双向绑定，view层的数据是有viewModel提供的，两者双向绑定，改变viewModel，view视图发生变化，网页手动改变视图view，viewModel发生变化，即所谓的数据双向绑定。

vue.js是MVVM模式中比较有吗的js框架了，它专注view层，核心是MVVM的VM,也就是viewModel,viewModel负责连接view和model，保证了视图和数据的一致性，这种轻量级的架构让前端开发变得更加高效便捷.

1、为什么会出现MVVM?

其实MVVM的兴起也正是不久前的15年，也正是vue.js刚出来的时候。在这之前，一个设计模式霸占着设计模式许久，没错那就是MVC，我最早听说的也是MVC，java中经典的SSH框架就用的MVC来构建的。之前有写过php，用的是php的YII2框架，它采用的也是MVC构建的前后端结合的项目。MVC即Model-View-Controller，就是模型-视图-控制器。

  • View 讲controller返回过来的数据展示给用户
  • Model 负责提供后端的数据，说白了就是数据库里的各种表
  • Controller 负责接收View的操作，对Model里的数据进行一顿操作，中间可能会进行 各种逻辑操作，最后再将操作后的model数据传给View层，展示给用户。

在HTML5还未火的那些年，mvc作为前端的最佳实践是ok的，这是因为当时view层还相对来说比较简单，前端需要的数据基本由后端处理好，View层主要获取到进行展示就行了。那时候提倡的是Controller来处理复杂的业务逻辑，所以View层相对来说就比较轻量，就是所谓的瘦客户端思想。

####但随着前端需求的不断提高，越来越多的技术接踵而来，要求高了，那前端的位子也变得更加重要，内容也变得更加丰满，因此前后端分离也成为了不可避免的一步，于是前端采用MVC架构，作为一个独立的工程项目来进行维护。

2.前端为何要工程化，用使用MVC？

相对HTML4，HTML5最大的亮点就是它为移动设备提供了一些非常有用的功能，使得HTML5具备了开发App的能力，HTML5开发App最大的好处就是跨平台，快速迭代和上线，节省人力成本和提高效率，因此很多的企业开始对传统的App进行改造，逐渐用H5代替Native，到15年的时候，市面上大多数App或多或少都嵌入了H5的页面或者讲H5打包陈App.既然要用H5来构建App，那View层要做的事情就不仅仅是数据展示这么简单了，不仅要管理复杂的数据状态，还要处理移动设备的各种操作行为，前端也需要工程化，需要一个类似MVC的框架来管理这些复杂的逻辑，使开发更加高效，这里的MVC稍微发生了点变化：

  • View Ui布局，展示数据。
  • Modal 管理数据
  • Controller 响应用户操作,并将Model更新到View上

这种MVC架构模式对于简单的应用来说还ok，也符合软件架构的思想。但实际上，随着H5不断发展，人们更希望用H5开发的应用都能和native媲美，或者接近原生App的体验效果，于是前端应用的复杂程度不同往日，今天的前端暴露了三个痛点：

  • 开发者再代码中大量调用相同的DOM API，处理繁琐，操作冗余，是的代码难以维护
  • 大量的DOM操作使页面渲染性能降低，加载速度变慢，影响用户体验
  • 当Model频繁发生变化，开发者需要主动更新到View；当用户的操作导致Model发生变化，开发者同样需要将变化的数据同步到Model中，这样的工作环境不仅繁琐，而且很难维护复杂多变的数据状态。

    其实，早期jQuery的出现就是为了前端更能简洁地操作DOM而设计的，但它再解决了第一个问题，另外两个问题始终伴随这前端。

3.MVVM 的出现，完美解决以上三个问题

MMVVM室友Model，View，ViewModel三部分组成，Model层代表数据模型，可以在Model中定义数据修改和操作的业务逻辑；View代表UI组件，他负责将数据模型转化成UI展现出来，ViewModel是一个同步View和Model的对象。

在MVVM架构下，View和Model没有直接的联系，二十通过ViewModel进行交互，Model和ViewModel之间的交互是双向的，因此View数据的变化会同步到Model中，而Model数据的变化也会立即反应到View上。

ViewModel通过双向数据绑定把View层和Model层连接起来，二View和Model之间的同步工作完全是自动的，无需人为干涉，因此开发者只需关注逻辑，不需要手动操作DOM，不需要关注数据状态的同步问题，复杂的数据状态维护完全有 由MVVM来统一管理。

4.Vue.js的细节

Vue.js可以说是MVVM架构的最佳实践，专注MVVM中的ViewModel，不仅做到了数据双向绑定，而且也是一筐相对轻量级的js库，api简介，容易上手。

Vue.js是采用Object.defineProperty的getter和setter，并结合观察这模式来实现数据绑定。当把一个普通Javascript对象传给Vue实例来作为data选项时，Vue将遍历它的属性，用Object.defineProperty将它们转化为getter/setter。用户看不到getter/setter，但是再内部它们让Vue追踪追踪依赖，在属性被访问和修改是通知变化。

  • Observer 数据监听器，能够对数据对象的所有属性进行监听，如有变动可拿到最新值并通知订阅者，内部采用Object.defineProperty的getter和setter来实现。

  • Compile 指令解析器，它的作用对每个元素节点的指令进行扫描和解析，根据指令模板替换数据，以及绑定相应的更新函数。

  • Watcher 订阅者， 作为连接 Observer 和 Compile 的桥梁，能够订阅并收到每个属性变动的通知，执行指令绑定的相应回调函数。
  • Dep 消息订阅器，内部维护了一个数组，用来收集订阅者（Watcher），数据变动触发notify 函数，再调用订阅者的 update 方法。

当执行 new Vue() 时，Vue 就进入了初始化阶段，一方面Vue 会遍历 data 选项中的属性，并用 Object.defineProperty 将它们转为 getter/setter，实现数据变化监听功能；另一方面，Vue 的指令编译器Compile 对元素节点的指令进行扫描和解析，初始化视图，并订阅Watcher 来更新视图， 此时Wather 会将自己添加到消息订阅器中(Dep),初始化完毕。

当数据发生变化时，Observer 中的 setter 方法被触发，setter 会立即调用Dep.notify()，Dep 开始遍历所有的订阅者，并调用订阅者的 update 方法，订阅者收到通知后对视图进行相应的更新。

1.找出重复的元素：这个思想就是先对数组进行一次排序，然后两两进行比较，然后借可以确定下来了。

解法1：

var arr = [4,3,2,7,8,2,3,1];
var newarr = [];
for(var i=0;i<arr.length;i++){
 arr.sort().sort(function(a,b){
   if(a===b&&newarr.indexOf(a)===-1){
       newarr.push(a)
   }
 })
}
console,log(newarr)

解法2：

var arr = [9,9,11,,2,3,4,5,6,77,7];

var sorted_arr = arr.slice().sort();

    results.push(sorted_arr[i]);

5.给定一个整数数组，其中有两项之和为一个特定的数字，不允许两次使用同一个元素，返回这两个数的索引。

比如：
给定nums = [2,7,11,15],target = 9
由于nums[0] + nums[1] = 9
所以返回[0,1]

function twoIndex(nums,target){
 var tmp = {};
 for(var i = 0;i<nums.length;i++){
  var num = nums[i];
  if(tmp[num]!==undefined){
   return [tmp[num],i]
  }else{
   tmp[target-num]=i;
  }
 }
}

function longestUniqueSubstr(str){
 str = String(str);
 var index = {}//存储索引
 var tenLen = 0//暂存结果
 var resLen = 0//最终结果
 for(let i = 0;i<str.length;i+=1){
  //取出索引值，如果有的话
  let ic =index[str[i]]
  if(typeof ic !== 'undefined'){
   //这里是关键部分
   if(tenLen > resLen){
    resLen = tenLen;
   }
   i = ic+1
   index = {}
   tenLen = 0
  }

  tenLen += 1
 }

浅谈事件轮询中的task与microtask

发表于 2017-04-14

浅谈事件轮询中的task与microtask

前言

想必大家对js的执行有很多不明白和困惑，今天我上网查了些许资料并根据自己的理解做一个总结。

我们先看下面这个代码
console.log('script start');
setTimeout(function(){
   console.log('setTimeout');   
},0);
Promise.resolve().then(function(){
  console.log('promise1');
}).then(function(){
  console.log('promise2');
});
console.log('script end');

这个代码的打印顺序是什么呢？
正确的结果是：script start,script end,promise1,promise2, setTimeout,不同的浏览器可能会有差异。
首先先要理解：
1.每一个‘线程‘都有它独立的时间轮询，所以每个页面都可以各自工作，执行它们自己的代码。所有一个来源的窗口都共享同一个时间轮询，彼此之间同步交流信息。事件轮询不断运转执行所有的任务队列。一个事件轮询中的任务可能来源于多个地方，需要保证 所有任务按正确的顺序执行并不简单，但是浏览器会帮忙选择如何执行这些任务。这样一来，浏览器可以对一些影响性能的操作（如用户输入）做特殊处理。
2.tasks已经被提前排好序，保证了浏览器可以持续从内部取出它们并弄到JS/DOM中执行。在两个任务的执行空隙，浏览器可能会重新渲染视图。在解析HTML页面的时候，鼠标点击事件与对应对调函数会产生一个新的task，同时会产生事件序列的还有上面的例子：setTimeout。
setTimeout会延迟指定的时间，然后将回调函数加入任务序列中。这就是为什么setTimeout会在script end后面打印。script end的打印属于第一个任务序列的一部分，而setTimeout则在下一个任务序列中被打印。
3.Microtask通常在js当前主任务执行完后直接执行，比如说对一些特殊事件作出响应，或者在不影响主线程情况下异步执行某些事件。一旦没有其他js代码在执行中，microtask队列会立即执行，执行过程中如果有microtask插入，也会同时被执行。microtask包括mutation observer回调，与上面例子中的promise回调。
一旦一个Promise被决议，在决议就会形成一个microtask来响应回调函数。这个可以保证promise的即使被决议，回调函数也会被异步执行。因此，调用then（rel，rej）方法后会立即生成一个microtask队列。这也就是为什么promise1和promise2在script end后面打印，microtask必须在当前js代码运转完后才被操作。promise1和promise2在setTimeout之前打印，也就是microtask永远在下一个task之前执行。

总结：所以，按我的理解，每个线程都有自己独立的时间轮询，轮询哪些代码是接下来该执行的，在这个线程里会有主代码和任务队列，主代码会立即执行，任务队列后执行，而任务队列又分为task和microtask，microtask通常在js当前主任务执行完后直接执行，也就是说一旦主代码没有内容了，microtask队列会立即执行。而task就会屁颠屁颠地跟在microtask后面。。。
